# `@appe/core`

The APPE estimator, with no framework attached: the models.dev catalogue, the
token estimators, the image/PDF/audio tokenization rules, and the pricing maths.
Pure TypeScript — it runs in a browser bundle and in plain node alike.

It is the single source of truth for what an estimate *costs*. The web app
(repo root) imports it, and the forthcoming `appe` CLI will import the same
functions, so the two can never disagree about a number.

## Usage

```ts
import {
  ALL_TEXT_MODELS,
  ALL_MODELS_BY_ID,
  computeTokens,
  computePrices,
  DEFAULT_APP_DATA,
} from "@appe/core";

const appData = {
  ...DEFAULT_APP_DATA,
  prompt: "Summarise this support ticket in one sentence.",
  example: "The customer cannot log in after the password reset.",
  dataCount: 10_000,
};

const model = ALL_MODELS_BY_ID["anthropic/claude-haiku-4-5"];
const tokens = computeTokens(appData, model);   // sync, rough tokenizer
const price = computePrices(appData, model, tokens);

console.log(price.totalCost); // $ for the whole 10k-item run
```

`computeTokens` is synchronous and uses the rough ~4-chars-per-token estimate;
`computeTokensAsync` uses the provider's real tokenizer where one exists
(js-tiktoken for OpenAI) and falls back to the rough one otherwise.

## What's in here

| Path | What |
| --- | --- |
| `src/data/` | The generated catalogue: `models.json`, `provider_data.json`, `models.meta.json` + the derived `ALL_*` selectors |
| `src/tokenization/` | `strToTokens` / `strToTokensSync` — text → token count, per provider |
| `src/imageCost.ts` | Provider image tiling rules (OpenAI 512-px tiles, Anthropic/Mistral px-per-token) |
| `src/computations.ts` | `computeTokens`, `computeTokensAsync`, `computePrices` — the estimator proper |
| `src/types/` | `Model`, `AppData`, `ProviderConfig`, `TokenResults`, `PricingResult` |
| `src/constants.ts` | Capability labels, audio defaults, the example templates |
| `src/__tests__/` | Vitest suite pinning every number above (`npm test` from the repo root) |

**Do not hand-edit `src/data/models.json`, `provider_data.json` or
`models.meta.json`** — they are generated by `scripts/sync-models.mjs` at the
repo root, which is run daily. Fix model data in the sync script's mapping (or
upstream at models.dev), never in the JSON.

## Consuming it

The package's `exports` point at TypeScript **source**, not a build artefact:
every consumer in this repo is TypeScript behind a bundler (Vite) or Vitest, so
a build step would only add drift. If `@appe/core` is ever published to npm on
its own, add a `tsup`/`tsc` build and repoint `exports` at `dist/` then.

Inside the repo it is wired up two ways, and both must stay in step:

- an npm workspace (`"workspaces": ["packages/*"]` at the root), so
  `node_modules/@appe/core` symlinks here;
- an explicit `@appe/core` alias in `vite.config.ts`, `vitest.config.ts` and the
  `paths` of `tsconfig.json` / `tsconfig.app.json`.
